# Contributing

Thanks for helping improve this project.

This repository is managed through GitHub Issues and pull requests. The goal is to keep changes focused, reviewable, and tied to a clear user outcome.

## Workflow

1. Start with a GitHub Issue.
2. Confirm the issue has a clear goal, scope, and acceptance criteria.
3. Work from a feature branch.
4. Open a pull request linked to the issue.
5. Document validation performed before requesting review.

## Issue Labels

Implementation work moves through these labels:

- `codex-ready`: ready for Codex or a contributor to work
- `codex-working`: actively being worked
- `codex-blocked`: blocked by missing information or failed validation
- `codex-review`: pull request is open and awaiting review
- `codex-done`: merged or completed

Only one issue should normally be marked `codex-ready` at a time.

## Pull Request Expectations

Every pull request should include:

- Summary of changes
- Linked issue
- Files or areas changed
- Validation performed
- Risks or follow-up work

## Development Standards

- Keep changes small and focused.
- Preserve existing architecture patterns.
- Reuse utilities and shared components when possible.
- Do not rewrite unrelated sections.
- Update release notes, changelog, or version metadata for user-visible changes.
- Prioritize mobile usability.
- Do not invent prices, cruise policy details, or unsupported claims.
